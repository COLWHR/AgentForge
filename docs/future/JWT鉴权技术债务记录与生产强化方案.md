# JWT鉴权技术债务记录与生产强化方案

## 文档目的

本文件用于完整记录当前项目中与 JWT 鉴权相关的技术债务，并明确未来在引入正式账号系统、进入生产环境阶段后，Auth 鉴权体系应如何整体强化。

## 当前阶段约束

- 当前阶段禁止改动现有 JWT 鉴权流程。
- 当前阶段必须保留 JWT 可绕过能力，且不能阻碍本地开发、联调、测试、E2E 验证与 Vibecoding 平台开发。
- 本文件只做债务记录与未来方案沉淀，不代表当前迭代执行任何 JWT 流程调整。

## 当前实现现状

当前 JWT 鉴权行为主要集中在以下位置：

- `backend/core/config.py`
- `backend/api/dependencies.py`
- `backend/main.py`
- `tests/unit/test_auth.py`
- `tests/unit/test_auth_dev_bypass.py`
- `tests/e2e/test_fail_paths.py`

当前实现的关键现实如下：

- 项目在开发环境下默认允许 `dev bypass` 生效。
- `resolve_auth_context` 在开发环境中可直接返回固定开发身份。
- 现有实现允许绕过 JWT，以确保本地开发与测试效率。
- 项目尚未引入正式账号系统、用户中心、会话体系、刷新令牌体系、密钥轮换体系与多端登录治理。
- 当前 Auth 更多是“开发期可运行机制”，而不是“生产期安全鉴权体系”。

## 当前 JWT 相关技术债务

### 1. 开发绕过与正式鉴权耦合

- 开发绕过能力与 JWT 正式校验逻辑位于同一主流程中。
- 开发环境默认行为会直接影响 JWT 分支是否真正执行。
- 当前设计更偏向“本地快速可用”，不适合直接外推为生产安全策略。

### 2. 鉴权开关语义不够清晰

- 当前 `dev/local/development` 环境下的鉴权行为由环境语义与 bypass 逻辑共同决定。
- “开发环境默认可绕过”与“显式启用绕过”不是完全分离的概念。
- 这会让测试、联调、平台运行时对 Auth 行为的预期不够稳定。

### 3. 身份来源模型仍然过于单一

- 当前主要围绕固定开发身份与基础 JWT 载荷做认证。
- 尚未接入真实账号系统、组织体系、用户目录与外部身份源。
- 缺少对用户冻结、禁用、注销、租户切换、会话失效等真实身份生命周期的支持。

### 4. 缺少完整的生产级令牌体系

- 仅有基础 JWT 解析与字段读取。
- 尚未建立 Access Token / Refresh Token 分层机制。
- 尚未建立会话续签、主动登出、单端失效、全端失效、设备管理等机制。

### 5. 缺少密钥治理与签名策略治理

- 当前未形成生产级密钥轮换方案。
- 尚未定义多密钥并行验证、过渡期轮换、历史 token 兼容窗口。
- 尚未定义面向未来的 JWKS、KID、外部 IdP 对接策略。
- 当前测试中已出现 JWT HMAC key 长度不足 warning，说明安全基线仍偏开发期。

### 6. Token 载荷模型未产品化

- 当前 token claim 仅覆盖基础用户与团队标识。
- 尚未定义标准化 claims 规范，例如：
  - 用户主键
  - 团队/组织主键
  - 角色与权限版本
  - 会话 ID
  - 设备 ID
  - 签发方
  - 受众
  - token 版本
  - 风险等级
- 缺少 claims 版本化策略，未来升级时容易引发兼容问题。

### 7. 认证与授权边界尚未彻底分层

- 当前流程中认证、成员校验、团队归属校验、资源访问校验存在相互邻接甚至交织。
- 尚未形成清晰的 AuthN / AuthZ / Tenant Boundary / Resource Policy 分层。
- 在未来账号系统接入后，如果不先分层，复杂度会快速上升。

### 8. 缺少正式 RBAC/ABAC 策略模型

- 当前角色字段存在，但整体仍偏轻量。
- 尚未形成正式角色矩阵、权限点、资源级授权策略。
- 尚未支持基于团队、资源归属、状态、环境的组合授权决策。

### 9. 缺少服务间鉴权与平台代理身份设计

- 当前重点仍是用户请求入口鉴权。
- 未来若引入云端编排、任务调度、远程执行、平台代理调用，需要服务到服务认证机制。
- 尚未定义内部服务令牌、平台代理身份、任务回调签名校验等方案。

### 10. 缺少审计、追踪与安全运营能力

- 当前已有基础日志，但仍不是完整审计体系。
- 尚未形成针对登录、鉴权失败、跨团队访问、权限拒绝、敏感资源访问的审计事件模型。
- 尚未建立安全告警、异常登录检测、token 滥用检测与风控扩展位。

### 11. 测试契约与当前开发策略存在偏移

- 现有部分测试期望 JWT 分支在开发环境下直接生效。
- 但当前项目现实约束是开发阶段允许甚至依赖绕过能力。
- 这导致测试预期与当前开发期策略并不完全一致。

### 12. 文档与运行语义未完全收口

- 当前项目尚未形成一份面向开发者的正式 Auth 行为说明。
- 缺少“开发期为何绕过、何时绕过、哪些环境绕过、未来何时收紧”的统一文档。
- 容易导致后续接手者误判当前行为是 bug、临时 hack，还是项目明确策略。

## 当前阶段必须明确保持不变的事项

以下事项在当前开发阶段必须保留，不进入本轮整顿实施范围：

- 保留现有 JWT 可绕过能力。
- 不收紧当前本地开发与测试环境的鉴权入口。
- 不以安全名义阻断开发、联调、E2E 与平台实验。
- 不在当前阶段引入账号系统、登录系统、刷新令牌体系、外部身份源集成。
- 不将当前整顿范围扩展为 Auth 重构。

## 未来引入生产环境账号系统时的全面强化目标

当项目进入生产阶段，并正式引入账号系统后，Auth 鉴权体系应达到以下目标：

- 认证链路稳定、明确、可审计。
- JWT 不再承担“开发便捷入口”与“正式生产认证”双重职责。
- 绕过机制仅限本地开发专用，并与生产部署物理隔离。
- 账号、组织、团队、角色、权限、会话、设备、服务身份形成统一模型。
- 所有权限校验具备清晰策略来源、清晰拒绝原因与完整审计记录。

## 未来生产 Auth 的推荐实现方式

### 一、身份体系层

- 引入正式账号系统，建立用户、组织、团队、成员关系的权威数据源。
- 明确 User、Organization、Team、Membership、Role、Permission 的领域模型。
- 将身份停用、封禁、退出组织、角色变更纳入实时鉴权结果。

### 二、认证层

- 采用标准 Access Token + Refresh Token 架构。
- Access Token 短周期有效，用于接口访问。
- Refresh Token 长周期有效，用于会话续签。
- 引入 Session 实体，用于支撑主动失效、单设备退出、全端登出与风控干预。

### 三、JWT 发行与验证层

- 统一 token 签发服务，禁止业务侧随意生成 JWT。
- 为 JWT claims 建立版本化规范。
- 引入 `kid` 与密钥轮换机制。
- 为将来的外部身份提供商对接预留 JWKS 能力。
- 明确 `iss`、`aud`、`sub`、`exp`、`iat`、`nbf`、`jti`、`sid` 等标准 claims 使用规范。

### 四、授权层

- 将认证与授权彻底拆分。
- 认证只负责“你是谁、属于哪个租户、持有哪些会话上下文”。
- 授权负责“你能否访问某类资源、能否执行某种操作、是否满足团队边界与角色策略”。
- 引入资源级权限决策，逐步形成正式 RBAC，必要时扩展到 ABAC。

### 五、开发绕过层

- 保留开发绕过能力，但与生产鉴权完全隔离。
- 明确仅在本地开发目标中启用，不进入生产构建与生产配置。
- 绕过能力应显式标注、显式配置、显式审计。
- 绕过身份、组织、角色应可配置，但必须带有清晰环境边界。

### 六、平台与服务鉴权层

- 为平台任务调度、代理执行、远程工作流回调设计服务到服务认证机制。
- 区分“用户身份”和“平台代理身份”。
- 对内部服务调用采用独立凭证体系，避免复用用户 JWT。

### 七、安全运营层

- 建立 Auth 审计日志模型。
- 建立异常登录、权限拒绝、跨租户访问、敏感操作的统一事件记录。
- 为将来的风控、封禁、会话撤销、密钥轮换演练提供基础设施。

## 未来生产 Auth 的推荐分阶段落地路径

### 阶段 A：账号系统落地前的设计冻结

- 冻结身份领域模型。
- 冻结 JWT claims 规范。
- 冻结 AuthN / AuthZ 分层边界。
- 冻结开发绕过与生产鉴权的环境隔离规则。

### 阶段 B：账号系统与会话体系接入

- 接入用户、组织、团队、成员关系。
- 上线会话模型。
- 上线 Access Token / Refresh Token 双令牌机制。

### 阶段 C：生产级授权模型接入

- 落地角色矩阵。
- 落地资源级授权。
- 落地租户边界与团队边界统一策略。

### 阶段 D：安全治理与平台治理

- 接入密钥轮换。
- 接入审计与告警。
- 接入平台内部服务身份体系。

## 未来生产阶段的验收标准

- 开发绕过不再影响生产鉴权语义。
- JWT 验签、会话状态、账号状态、团队成员关系、角色权限全部纳入统一决策。
- 所有关键鉴权失败都有明确错误码、明确日志、明确审计记录。
- 账号退出、封禁、角色变更、会话失效可以实时生效。
- 平台内部调用与用户调用的身份边界清晰分离。

## 当前整顿阶段与本文件的关系

本轮项目整顿只将 JWT 相关内容作为技术债务记录，不纳入实施改造范围。

本轮整顿明确排除以下事项：

- 不修改现有 JWT 逻辑分支。
- 不修改开发环境默认可绕过的现实策略。
- 不收紧当前鉴权限制。
- 不引入生产级账号系统。
- 不启动 Auth 重构。

## 后续执行原则

- 当前阶段优先保障开发连续性。
- 未来生产阶段再整体升级 Auth，而不是在开发阶段零散收紧。
- 所有 Auth 强化动作必须以账号系统正式引入为前提统一推进。
